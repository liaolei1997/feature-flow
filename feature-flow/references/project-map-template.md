# project-map.md 模板与写作原则

project-map 是**骨架级**的项目认知，每次 `/feature-flow` Phase A 自动加载。目的：让新一次开发**不需要从头描述项目**。

## 核心原则

1. **定源**：分两类内容，来源不同——
   - **结构层**（业务功能 / 技术栈 / 模块地图 / 领域概念）是**当前代码的投影**，由「更新」/P0 漂移**从代码重新生成**，不靠 session 累积。
   - **经验层**（关键约定 / 非显而易见之处 / 为什么）是**代码看不出、且稳定耐久**的认知，由开发沉淀，但只记长期成立的。
   - **永不写**："本次刚实现/在途"的设计、行号、函数级调用链——这些会被推翻或随代码漂移，属于 session 文档。
2. **要"怎么做的"，不要行号**：核心功能必须写清**概念级主流程 + 关键机制**（够下次开发不读代码就懂大概怎么跑）——这是有用信息；但**不写行号、不写函数调用链、不贴大段代码**——那些一改就过期。区别是"机制/流程"（稳定）vs"实现坐标"（易腐）。
3. **面向非显而易见**：看代码能直接看出来的样板少写；看不出来的（为什么 X 必须走 Kafka、某字段的真实含义）必须写。
4. **详略有别**：核心业务（二）写厚、把"怎么做的"讲透；其余节精简。全文目标 < 250 行。

## 模板

```markdown
# Project Map — <PROJECT_ID>

> 骨架级项目认知，给 /feature-flow Phase A 加载用。
> 最后更新：YYYY-MM-DD

## 一、项目概述
- 一句话定位：<这个项目是干啥的>
- 当前阶段：<MVP / 线上 / 迭代中 / 重构中>
- 项目根路径：<本机 cwd>

## 二、核心业务 / 功能地图
> 这是 map 最有价值的一节，要写**够厚**。每个核心能力写清三件事：
> ① **是什么**——用户能干啥、解决什么问题；
> ② **怎么做的**——主流程（3–6 步：触发 → 关键步骤/机制 → 产出）+ 用到的关键模型/数据结构/算法（概念级）；
> ③ **落在哪**——主要模块/文件（不写行号、不写函数调用链）。
> 标准：看完不读代码也能讲出"这个功能大概怎么跑的"。只写一句话价值 = 不合格。

- **<功能名>**：<一句话价值>。
  - 怎么做：<触发> → <关键步骤1> → <关键步骤2/核心机制，含用到的关键数据或模型> → <产出>。
  - 落在：`<模块A>` / `<模块B>`
- **合同上传与解析**：用户传合同 docx，系统抽出可定位的条款结构供后续审查。
  - 怎么做：两条上传链路（前端 WebOffice 编辑器 SDK + WPS 365 开放平台云盘）→ 文件存 KS3 → 解析 docx 提取每段 `para_id` 与段落/句子结构 → 落库，作为审查与卡片定位的锚点。
  - 落在：`services/upload_service.py` / `services/wps/` / `services/document_intelligence/`
- **条款风险审查（CR）**：对合同逐条跑规则 + LLM，产出带定位的审查卡片。
  - 怎么做：进审查 → 按合同类型选清单规则 → 每条规则走 `rule_engine` 统一调 LLM（段级/全文级筛选 + 单规则审查）→ 生成 RiskItem/审查卡片 → 异步 reconcile 把卡片按 `para_id` 锚点对齐到最新文档位置。审查 turn 已从传统表单切到对话引擎。
  - 落在：`services/rule_engine.py` / `scheduler/executors/`（reconcile）/ `graph/`（对话引擎）

## 三、技术栈与运行
- 语言/框架：...
- 数据库/中间件：...
- 本地启动：<命令>
- 测试：<命令>
- 入口文件：<路径>

## 四、模块地图
> 只列顶层模块 + 一句话职责；子模块按需展开 1 层。

- `module-a/`：做 X，对外暴露 Y
- `module-b/`：...
- 关键依赖关系：A → B（同步）；B → C（异步 Kafka）

## 五、核心领域概念
> 新人最容易懵的业务术语，每个一句话 + 代码位置。

- **Annotation**：审查批注，一条对应合同中一段风险。见 `schemas/annotation.sql`
- **RiskItem**：批注聚合成的风险项，一个 RiskItem 可含多 Annotation。见 `models/risk.py`

## 六、关键约定（项目特有）
> 不写代码看不出来的约定；写看了代码也搞不清楚"为什么这样"的事。

- **错误码规范**：所有 4xx 返回 `{code, message, detail}`，code 在 `errors/codes.py`。不直接 raise HTTPException。
- **DB 时间字段**：全部 UTC，命名 `created_at/updated_at`，禁用 `_time` 后缀。
- **LLM 调用**：走 `llm/gateway.py`，不直连 SDK。gateway 做了限流和审计。

## 七、非显而易见之处（坑 & 反模式）
> 踩过的坑 / 看着合理但不能这么干的地方。

- `review_service.py` 里的 `batch_size` 表面是批量大小，实际是**每批 LLM token 预算**，改它要算 token。
- 历史遗留：`old_annotations` 表只读保留，写入请走 `annotations`。迁移日期 2025-09-01。

## 八、外部依赖与边界
- LLM：<厂商，模型，限流>
- OSS：<bucket，路径约定>
- 第三方 API：<名字，文档链接>

## 九、最近变更（last_synced_commit: n/a @ <date>）
<!-- 标题括号内由 P8 回写为 last_synced_commit: <sha7> @ <date>，供 P0 漂移检测；最多保留 10 条 -->
- 2026-04-24 <feature-slug>：一句话摘要（≤30 字）
```

## 首次建立 map 的流程（命令 Phase A.3 执行）

1. `Agent(subagent_type=Explore, thoroughness=medium)` 扫项目根：
   - 识别语言/框架/依赖
   - 识别顶层模块
   - 识别入口（main / app / server）
   - 读 README / CLAUDE.md（如有），据此初拟「核心业务/功能」
   - 填充上表 1-4 节（概述/业务功能/技术栈/模块地图），5-8 节留空待用户补

2. 用 `AskUserQuestion` 向用户补 3-4 个关键问题（一次问完）：
   - "确认/补全 2-3 个**核心业务功能**（这产品主要让用户干啥）"
   - "补 2-3 个**核心领域概念**（新人最容易混的业务术语）"
   - "补 1-2 个**项目特有约定**（看代码也看不出为啥这样做的规矩）"
   - "有没有踩过的坑 / 反模式？"

3. 生成首版 `Write` 落盘

## 更新 map 的判断（命令 P8.2 执行）

开发完只沉淀**代码看不出、且稳定耐久**的经验：

| 本次是否… | → 更新小节 |
|---|---|
| 定下并会长期沿用的新约定（命名/错误码/DB 模式） | 六、关键约定 |
| 踩到/确认了坑、或"试过 X 已回退别再来" | 七、非显而易见之处 |
| —（每次必写） | 九、最近变更 追加一行 |

**结构层（二业务 / 三技术 / 四模块 / 五概念）P8 不手写**——它是代码投影，由「更新」从代码重新生成。结构有实质变化只在「九、最近变更」记一行，下次刷新自然反映。

**啥都不写进经验层**的情况：
- 单纯改 bug / 调参数 / 加日志
- 实现细节变化（"函数从 for 改成 map-reduce"）
- 本次刚实现、还可能被推翻的设计

这些进 session，map 不关心。
