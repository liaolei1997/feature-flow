# 拷问规范（详版）

> 本文件是"Phase 3 拷问"方法论的详细版，含好/坏例子。
>
> **注意**：主命令 `commands/feature-flow.md` 已内联核心规范用于注入 feature-dev 的 args。本文件供：
> 1. 运行时 Claude 可选 Read，获取更详尽示例和判定口诀
> 2. 人工维护方法论时的主文件（更新此处再同步到主命令）

## 核心视角与职责边界

**视角**：站在"我要开始实现这个需求"的**开发者视角**，把动手前必须解决的拦路问题挖出来。不是挑 PRD 的错。

**边界（关键！）**：Phase 3 拷问**只问业务层面（PRD-TBD）**，不问工程选型。

| 问题性质 | 归属 |
|---|---|
| 业务规则 / 阈值 / 优先级 / 行为定义 | **Phase 3 拷问**（[PRD-TBD]） |
| 工程选型（同步 vs 异步、新表 vs 旧表、错误码、表结构、缓存） | **Phase 4 Architecture Design**（code-architect 方案对比） |

**为什么这么分**：feature-dev 的 Phase 4 会用 2-3 个 code-architect agent 并行给出完整方案对比。如果 Phase 3 再单点问工程选型，用户被双重打扰。方案对比形式比单条拷问信息量更大，用户看完整方案再拍板更明智。

## 代码驱动（Phase 2 阶段，强制前置）

Phase 3 每条问题的"依据"字段必须引用代码事实（`file:line`）。这些事实由 **feature-dev 的 Phase 2** 产出——args 里注入的"Phase 2 额外指令：PRD 假设验证"要求 Phase 2 除了默认架构探索外，额外做：

1. 从 PRD 抽出所有具名实体：接口名 / 字段名 / 表名 / 枚举 / 状态机 / 外部服务名
2. 每个实体用 `Grep` + `Read` 定位真实代码；找不到的记为"PRD 新增"或"PRD 术语错用"
3. 核对现有接口入参/出参、枚举实际取值、已有校验、DB 字段和索引，与 PRD 假设比对
4. 产出"PRD 假设验证摘要"（3-8 行，供 Phase 3 用）

Phase 3 拷问直接消化这份摘要。

## 8 镜头（每面镜下只识别 PRD-TBD）

每面镜子下逐条扫描，**只把业务层面的缺口登记为问题**；发现工程选型岔路口的，记入 prd-check.md 的"工程选型线索"章节（给 Phase 4 参考），不向用户提问。

### 1. 逻辑完整性
**问业务**：流程是否自洽？前后因果闭合？隐藏前提？规则冲突时谁赢？
**不问工程**：模块间调用时序（那是 Phase 4 事）

### 2. 用户操作路径
**问业务**：实际触发时机？并发时的产品期望？能否撤销/重做？撤销后数据的**产品语义**（显示/隐藏/删除）？

### 3. 边界条件
**问业务**：入参/数量/时间/权限边界下，**产品期望的行为定义**（报错文案？降级？）

### 4. 限制与配额
**问业务**：硬上限数值（"最多批量多少份"）、触顶时的**产品侧响应**（显示限额提示？入队？）
**不问工程**：具体限流算法、Redis 还是 Semaphore（Phase 4）

### 5. 异常与错误处理
**问业务**：失败后**用户看到什么**、回滚到什么状态、提示文案方向
**不问工程**：错误码分配（1001 还是 2001）、retry 策略参数（Phase 4）

### 6. 新数据结构
**问业务**：要不要存这个字段？字段必填吗？是否面向用户可见？
**不问工程**：表结构、字段命名、索引（Phase 4）

### 7. 旧数据迁移
**问业务**：历史数据要不要回填到新规则？回填不了的怎么向用户说明？新老功能共存多久？
**不问工程**：回填脚本实现、批次大小（Phase 4）

### 8. 权限 / 可见性 / 合规
**问业务**：谁能触发这个操作（角色 / 归属）？各角色、各参与方分别能看到什么、看不到什么（可见性）？这个改动要不要留审计痕迹（谁 / 何时 / 改了什么）？涉及的敏感数据（合同正文 / PII / 当事方信息）有无访问、留存、脱敏方面的业务要求？
**不问工程**：鉴权中间件、RBAC 表结构、加密算法、审计日志怎么落库（Phase 4）
> 法律/合同场景"谁能看、谁能改、留不留痕"漏问代价最高。**涉及敏感数据或多角色访问时，本镜头问题升 P0。**

## 优先级映射

| 优先级 | 对应镜头 | 理由 |
|---|---|---|
| **P0** | 逻辑完整性 / 用户操作路径 / 新数据结构（业务语义） | 逻辑与设计是根基 |
| **P1** | 边界条件 / 异常与错误处理 / 限制与配额 / 权限·可见性·合规 | 健壮性与合规，前置最好 |
| **P2** | 旧数据迁移 | 兼容性可分批 |

具体问题可按阻塞度升降级。**权限·可见性·合规镜头：涉敏感数据/多角色访问时升 P0。**

## 写法模板

```
<n>. [P0|P1|P2][PRD-TBD][<镜头>] <一句话问题>
   依据：<代码位置 / PRD 段落>
   候选答案：A) ... / B) ... / C) ...
```

## 好例子（PRD-TBD）

**P0 / 用户操作路径**：
```
1. [P0][PRD-TBD][用户操作路径] 用户在批量审查进行中点击"取消"，已审完的风险项是否保留？
   依据：src/batch/runner.py:120 当前无取消逻辑；PRD §2.4 未定义取消语义
   候选答案：A) 全部丢弃（含已审完的）/ B) 保留已审完的，未审完不算数 / C) 提示用户选择
```

**P1 / 限制与配额**：
```
5. [P1][PRD-TBD][限制与配额] 批量审查上限多少份？超限时产品侧响应？
   依据：src/llm/gateway.py:78 LLM 每分钟 60 次限制；PRD §3 未定义业务上限
   候选答案：A) 硬限 50 份，超限直接报错 / B) 硬限 50 份，超限自动拆批异步 / C) 无业务上限，只提示预估时长
```

**P2 / 旧数据迁移**：
```
8. [P2][PRD-TBD][旧数据迁移] 2025 年前的审查记录没有"风险等级"字段，要不要回填？
   依据：schemas/annotations.sql:15 risk_level 字段 2026-01 后新增；PRD 未提旧数据
   候选答案：A) 不回填，老记录显示"未知等级" / B) 回填为默认"中"等级 / C) 异步重跑一遍 AI 打标
```

**P0 / 权限·可见性·合规**：
```
9. [P0][PRD-TBD][权限/可见性/合规] 批量审查结果，非发起人的同 workspace 成员能否查看？外部律师角色呢？
   依据：src/auth/acl.py:30 现按 contract.owner_id 鉴权；PRD §2 未提多角色可见性
   候选答案：A) 仅发起人可见 / B) 同 workspace 成员可见 / C) 按合同参与方分别可见
```

## 坏例子（禁止）

```
1. 建议进一步确认批量审查的具体实现方式。              ← 废话，没依据也没候选
2. 可能需要考虑并发问题。                              ← "可能"、无场景、无方案
3. 边界条件待补充。                                    ← 没写问题是什么
4. [PRD-TBD][新数据结构] batch_tasks 表应该建哪些字段？ ← 工程选型！应交 Phase 4
5. [PRD-TBD][异常处理] 重试用指数退避还是固定间隔？    ← 工程选型！应交 Phase 4
```

## 交互澄清回填格式

用户答后，在问题下追加：

```
1. [P0][PRD-TBD][用户操作路径] 用户取消时已审完的风险项是否保留？
   **答**（2026-04-24）：选 B。理由：保留已审完的对用户更有价值，未审完的重算成本低。
```
